## Spacing Scale

All values are multiples of 4. Tailwind spacing token shown alongside px value.

| Token | px | Tailwind class | Usage |
|-------|----|----------------|-------|
| xs | 4px | `gap-1`, `p-1` | Icon gaps, badge inner padding |
| sm | 8px | `gap-2`, `p-2` | Inline element gaps, tight stacks |
| md | 16px | `gap-4`, `px-4`, `py-4` | Default card padding, nav link gaps |
| lg | 24px | `gap-6`, `px-6` | Section horizontal padding, grid gutter |
| xl | 32px | `gap-8`, `mt-8` | Card grid gap |
| 2xl | 48px | `py-12` | Mobile section vertical padding |
| 3xl | 96px | `py-24` | Desktop section vertical padding |

Exceptions:
- Navbar height: `h-16` (64px) — preserves existing shell
- Touch target floor: `min-h-[44px] min-w-[44px]` on hamburger button (NAV-02 accessibility floor)
- Max content width: `max-w-content` (1200px, registered token) with `px-6` container padding

Source: DESIGN.md ("Section padding: 96px vertical, 48px mobile"), existing Navbar.tsx (`px-6`, `h-16`)

## Accessibility

### Hamburger Menu (NAV-02)

Required aria on the toggle button:
- `aria-expanded={isOpen}` — programmatic open state
- `aria-controls="mobile-menu"` — references menu panel id
- `aria-label` — dynamic: `"Open menu"` / `"Close menu"`

Required on the menu panel:
- `id="mobile-menu"` — target of `aria-controls`
- `role="navigation"` or wrapping `<nav>` with `aria-label="Mobile navigation"`

Focus management:
- When menu opens: move focus to first nav link inside the panel (`useEffect` + `ref.focus()`)
- When menu closes: return focus to hamburger button
- `Escape` key closes menu and returns focus to hamburger
- Tab order within open menu: hamburger → nav links (Solutions, Work, About) → language switcher pills

### Nav Links Active State (NAV-01)

- `aria-current="page"` on the active link: `aria-current={isActive ? "page" : undefined}`

### Hero CTA (HOME-01)

- Button or anchor with clear label. If `<a href="mailto:...">`: `aria-label` not required (link text is sufficient).
- Volt Green on near-black background (`#0a0a0a`): contrast ratio of `#96D02C` on `#0a0a0a` is ~7.4:1 — passes WCAG AA for normal text and large text.

### Solution Cards (HOME-02)

- If cards are not links: section is presentational, no additional aria needed.
- If cards are links: use `<a>` wrapping the card content; ensure link text is unique per card.

### Focus Ring Standard

All interactive elements use `focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-cyber-jade`. Never suppress focus outlines. Use `focus-visible:` (not `focus:`) to avoid showing ring on mouse click.

### Color Contrast Checks

| Element | Foreground | Background | Ratio | WCAG |
|---------|------------|------------|-------|------|
| Body text | `#ffffff` | `#0a0a0a` | 19.6:1 | AAA |
| Secondary text | `#a1a1aa` | `#0a0a0a` | 5.9:1 | AA |
| Muted text | `#52525b` | `#0a0a0a` | 2.7:1 | Fails AA — use only for non-essential labels |
| Cyber Jade on dark | `#008684` | `#0a0a0a` | 4.6:1 | AA (large text only) |
| Volt Green on dark | `#96D02C` | `#0a0a0a` | 7.4:1 | AAA |
| Nav links default | `#ffffff` | `#1a1a1a` | 16.7:1 | AAA |

Note: `text-muted` (`#52525b`) fails AA for body text — restrict it to copyright line, tagline, and decorative labels only (matching existing Footer.tsx usage).

## CSS Dot-Grid Background (HOME-01)

The hero section dot-grid is implemented in pure CSS/Tailwind — no images, no SVG sprites.

Implementation approach (in `globals.css` or via Tailwind `bg-` utility with arbitrary value — check Phase 1 arbitrary-value policy first):

```css
/* In globals.css — add .dot-grid class */
.dot-grid {
  background-image: radial-gradient(circle, #2a2a2a 1px, transparent 1px);
  background-size: 24px 24px;
}
```

Apply as `className="dot-grid"` on the hero section wrapper. The dot color (`#2a2a2a` = `border-subtle`) keeps the grid subtle against `#0a0a0a` background. No Tailwind arbitrary value required — single CSS class definition in globals.css is clean and static-export safe.

Hero wrapper must also set `bg-[#0a0a0a]` (or `bg-background` once token registered in Phase 1) so the dot-grid has the correct base color behind it.

## Implementation Notes

### Active Route Detection (NAV-01)

```tsx
"use client";
import { usePathname } from "next/navigation";

// Inside Navbar:
const pathname = usePathname();

// On each link:
const isActive = pathname === link.href || pathname.startsWith(link.href + "/");

// Class merge:
className={`text-sm font-medium transition-colors duration-150 ease-out ${
  isActive
    ? "text-cyber-jade border-b border-cyber-jade pb-1"
    : "text-white hover:text-cyber-jade"
}`}
// aria:
aria-current={isActive ? "page" : undefined}
```

Note: After Phase 1, routes are `app/[locale]/...` so `pathname` will be `/en/solutions`, not `/solutions`. The `href` values in `NAV_LINKS` must be updated to include locale prefix, or use relative matching (`pathname.endsWith(link.href)`). Coordinate with Phase 1 output.

### Hamburger State (NAV-02)

```tsx
const [isOpen, setIsOpen] = useState(false);

// Close on Escape
useEffect(() => {
  const handler = (e: KeyboardEvent) => {
    if (e.key === "Escape") setIsOpen(false);
  };
  document.addEventListener("keydown", handler);
  return () => document.removeEventListener("keydown", handler);
}, []);
```

Hamburger icon: use two states — 3-line SVG (closed) / X SVG (open). Both 20×20px inside the 44×44px touch target wrapper.

### Locale-Aware hrefs

Phase 1 establishes `app/[locale]/` routing. In Phase 2, all internal `href` values must be locale-prefixed. Options:
- Use a `useLocale()` hook from next-intl and prefix hrefs: `/${locale}/solutions`
- Or use next-intl's `<Link>` component which handles prefixing automatically

Use next-intl's `Link` import (`import Link from "next-intl/navigation"`) after Phase 1 sets up the navigation helpers. If Phase 1 does not export a typed Link, fall back to manual `/${locale}${path}` construction.

### Static Export Compatibility

- `Navbar.tsx` becomes `"use client"` — compatible with static export
- `usePathname()` is a client hook — safe for static export (resolves at runtime in browser)
- No server components with data fetching in this phase
- `new Date().getFullYear()` in Footer.tsx: move to client component or hardcode year if static export renders it at build time (current Footer.tsx already does this — acceptable for v1)
